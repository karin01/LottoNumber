# 작업 기록: 주간 당첨 요약이 최신 회차로 안 갱신되던 문제

**날짜:** 2026-04-13

## 현상

- 토요일 추첨(예: 4월 18일) 이후에도 화면에 **이전 주 회차**(예: 1219회, 4월 11일)만 표시됨.
- 데이터 출처가 `뉴스 RSS(추정)`일 때 특히 그런 경향.

## 원인 (WHY)

1. **동행복권 JSON API**가 호스팅(Render 등)에서 막히면, 주간 요약은 **로컬 `lotto_history.json`의 가장 큰 회차**만 사용했음 → 파일이 1219까지면 항상 1219.
2. 뉴스 RSS 조회 시 `draw_no=None`이면 **역시 파일 최대 회차**로만 검색해서, 1220회 기사를 찾지 않음.
3. `_guess_current_draw_no()`가 **서버 로컬 날짜(UTC)**만 쓰면 한국 토요일/일요일 경계에서 주차가 어긋날 수 있어, **KST 기준 날짜**로 통일함.

## 수정 요약

- `lotto_data.get_latest_draw_no()`: 동행복권 **메인 HTML**의 최신 회차를 우선 사용(가능할 때).
- API 실패 후 표시 회차: `공식 메인 최신`이 있으면 그 회차, 없으면 `max(로컬 최대, KST 추정 회차)`.
- `_fetch_weekly_summary_from_news`: 위에서 정한 회차로 RSS 검색 + `parse_round_from_news_text`로 **제목 속 당첨번호** 보강.
- `_fetch_lotto_raw`: `REQUEST_HEADERS`로 API 차단 완화 시도.
- 메인 파싱도 실패·번호 확보 불가 시에만, 예전처럼 **파일 최신 회차**로 폴백(빈 응답 방지).

## 추가 (2026-04-13 이후)

- `get_latest_draw_no()`(동행 메인 HTML)이 **여전히 1219**만 돌려주는 환경이 있어, `target_no`를 **공식값만** 쓰면 1219에 고정됨.
- 해결: `_expected_latest_announced_draw_no_kst()`로 **KST·토요일 21:15 기준 ‘발표된 마지막 회차’**를 구해 `target_no = max(로컬, 공식, 달력)`으로 병합.

## lt645 공식 추첨결과 API (추가)

- `https://www.dhlottery.co.kr/lt645/result` 페이지는 브라우저용이며, 실제 데이터는 **`selectPstLt645Info.do?srchLtEpsd=회차`** JSON으로 내려옴.
- `common.do?method=getLottoNumber`이 IP에서 막혀도 위 API는 동작하는 경우가 있어, `lotto_data._fetch_one_draw` → `fetch_lotto_draw_raw`에서 **폴백**으로 연동함.

## 크롤링·재시도 보강

- lt645: **재시도**(기본 3회, `REQUEST_DELAY_SEC` 간격), **`srchDir=latest`**로 최신 한 건과 요청 회차 일치 시 사용, **`/lt645/result` HTML 선요청** 후 API 재시도(브라우저 순서 유사).
- `get_latest_draw_no`: 메인 HTML 대신 **`srchDir=latest` JSON의 `ltEpsd`** 우선(지연·캐시로 메인이 1219에 머물 때 대비).
- `getLottoNumber`: 2회 재시도, lt645 전체 실패 후 페이지 워밍 뒤 **한 번 더** getLottoNumber 시도.

- `server.py`: `_get_weekly_summary`, `_fetch_weekly_summary_from_news`, `_guess_current_draw_no`, `_expected_latest_announced_draw_no_kst`, `_fetch_lotto_raw`
